# Generation Context Assembly Order

## Purpose

Assemble context in a stable order before generating prose. This method adapts
Novel-OS, NovelForge, Agnai memory books, SillyTavern character cards, and Story
Skills continuity checks into a VN-specific pre-draft stack.

## Borrowed Structure

- Novel-OS: separate durable standards, project-level decisions, and manuscript
  tasks so style rules do not mix with one-scene memory.
- NovelForge: use schema-first cards, context injection, knowledge graph ideas,
  and review cards to keep long-form generation controllable.
- Agnai memory books: retrieve only relevant memories by keywords, priority,
  and budget rather than stuffing every fact into the prompt.
- SillyTavern / Character Card V2: keep description, personality, scenario,
  examples, post-history instructions, and lorebook-style notes as separate
  control surfaces.
- Story Skills: treat continuity, promises, payoffs, questions, object state,
  and character appearances as deterministic contracts.

## Assembly Stack

Load in this order before drafting:

1. Workbench standards:
   - method cards
   - hard style stops
   - textbox and display rules
2. Project memory:
   - premise
   - theme spine
   - style rules
   - route map
   - interactive design
3. Manuscript task:
   - scene card
   - state delta
   - target output path
   - declared corpus metrics
4. Character control:
   - stable character cards
   - runtime state
   - memory logs
   - user-visible character brief if characters are new or revised
5. Retrieval:
   - only memory blocks with matching thread/object/character tags
   - promises and debts touched by the scene
   - current highest emotional axis for each speaker
6. Draft constraints:
   - choice contract
   - branch graph contract
   - row type contract
   - no project-only memories in reusable core
7. Post-draft checks:
   - validate
   - deep-audit
   - readable preview
   - engine exports
   - metric comparison

## Output Contract

`draft-session` or its equivalent must show:

- which method cards were loaded
- which project files were loaded
- which character memory blocks were relevant
- which output file is allowed to contain generated prose
- which QA and export commands must run after drafting

## Failure Signs

- A scene is written straight from chat context without a draft-session.
- Stable writing methods are saved inside a one-off project folder.
- Character-only facts leak into reusable skills.
- A character has a card but no runtime state or memory block.
- The final script cannot be regenerated by another Codex from local files.
